Part 1

The Manual Way

Six steps from zero to a live website.

GitHub Pages is free static website hosting. It serves HTML, CSS, JS and images straight from a repository with instant HTTPS, supports custom domains, and redeploys automatically on every push. Your URL will look like https://username.github.io/repo-name.

1

Create your HTML file

Add a file named index.html — GitHub Pages serves this as your homepage.

2

Init a local Git repo

git init creates a hidden .git folder, git add . stages files, and git commit saves a snapshot.

3

Create a GitHub repo

Use the github.com UI, or one command with the gh CLI.

4

Push your code

Connect your local repo to GitHub as origin and push to main.

5

Enable GitHub Pages

In Settings → Pages, deploy from branch main / root — or enable it via gh api.

6

Your site is live 🎉

First deploy takes ~60s. Watch the Actions tab; every push auto-redeploys.

⚠️
The file must be named exactly index.html (all lowercase). Index.html, home.html, or index.htm won't be served automatically.
Initialize & commit
# Navigate to your project folder
cd ~/projects/my-site
# Initialise a new git repository
git init
git add .
git commit -m "Initial commit"
Create & push (gh CLI)
gh auth login
# One command creates the repo + pushes
gh repo create my-site \
  --public --source=. \
  --remote=origin --push
Push manually & enable Pages
git remote add origin https://github.com/YOU/my-site.git
git branch -M main
git push -u origin main

# Enable Pages via the API
gh api -X POST /repos/YOU/my-site/pages \
  -f source[branch]=main -f source[path]=/
The development cycle
# edit, preview in browser, repeat...
git status              # what changed
git diff                # exactly what changed
git add --all
git commit -m "Added a pretty icon to the home page"
git push                # publish to your live site
🔧
Common gotchas: wrong filename · private repo needs GitHub Pro/Team · first deploy takes ~60s · 404s usually mean the wrong branch/folder in Settings → Pages · auth errors? run gh auth login.
Part 2

Automate It with a Claude Code Skill

One command. All six steps. Done.

A Claude Code skill is a custom /slash-command you define as a Markdown file in ~/.claude/commands/ (user-wide) or .claude/commands/ (per-project). When invoked, Claude reads the file and follows its instructions — running shell commands, editing files, and calling APIs. No plugin system, no build step.

Install the skill
mkdir -p ~/.claude/commands
curl -O https://raw.githubusercontent.com/\
louisroehrs/github-pages-training/refs/\
heads/main/deploy-github-pages.md
cp deploy-github-pages.md ~/.claude/commands/
Use it from any project
# In Claude Code, just type:
/deploy-github-pages my-landing-page

Claude checks prerequisites, creates the repo, pushes your code, enables Pages, and returns your live URL — all without leaving your editor.

deploy-github-pages.md — the skill file
---
description: Deploy the current project to GitHub Pages
---
Deploy this project to GitHub Pages.
Repo name (optional): $ARGUMENTS

1. Verify git is installed. If not, tell the user and stop.
2. Verify gh is installed and authenticated (gh auth status).
3. If no .git folder: git init, git add ., git commit.
4. If no remote 'origin': gh repo create $ARGUMENTS --public
   --source=. --remote=origin --push, then skip step 5.
5. Otherwise: git push -u origin main
6. Enable Pages: gh api -X POST /repos/{owner}/{repo}/pages
   -f source[branch]=main -f source[path]=/ (ignore 409).
7. Poll gh api /repos/{owner}/{repo}/pages --jq .html_url
   until it returns a URL. Print: 'Live at <url>'
1

Check prerequisites

Verifies git and gh are installed and gh is authenticated.

2

Git init if needed

Runs init, add, commit if no repo exists yet.

3

Create GitHub repo

gh repo create — public, sets the origin remote.

4

Push code

git push -u origin main.

5

Enable Pages

API call via gh api to activate Pages on main / root.

6

Return live URL

Polls until the site is live, then prints the URL.

⚡
Manual: 6 steps, ~5 minutes.  Automated: /deploy-github-pages, ~30 seconds — works in any project.
Part 3

Fork, Edit & Pull Request

Contribute lasting changes to any project — the right way.

A fork is your personal copy of someone else's repository, living under your GitHub account with full write access. The original is called the upstream. Your changes don't affect upstream until a Pull Request is reviewed and merged.

1

Fork the repository

Click Fork in the top-right on GitHub. Your fork is fully independent — pushes go to your copy only.

2

Clone your fork

Clone your fork (not the original), then add the original as upstream to pull future updates.

3

Branch, change, commit

Work on a dedicated branch — never commit directly to main.

4

Push & open a PR

Push the branch to your fork, then open a Pull Request against upstream.

Clone & track upstream
git clone https://github.com/YOU/hackerdojo.org
cd hackerdojo.org
git remote add upstream \
  https://github.com/hd-admin/hackerdojo.org
git remote -v
Branch & commit
git checkout -b fix/update-homepage-text
# edit index.html ...
git add index.html
git commit -m "Update homepage welcome text"
Push your branch & open the PR
git push -u origin fix/update-homepage-text

gh pr create --repo hd-admin/hackerdojo.org \
  --title "Update homepage welcome text" \
  --body "Improves the welcome message for new visitors."
After your PR is merged — stay in sync
git pull upstream main   # latest onto your machine
git push origin main     # update your fork on github.com
💡
Branch names like fix/…, feat/…, or docs/… signal intent at a glance and keep your main branch clean. Always clone your fork — cloning upstream gives you no push access.

You're live. 🚀
